## What shipped
Hardening patch on top of the previous run's primary fix (`6b9c548`, which
reduced the hero carousel from 14 to 5 cards and stopped the animation
collision). The primary cause was fixed, but the task was still `Todo` and
the bug could silently regress — so this run made the fix load-bearing:

1. **`HERO_CAROUSEL_MAX = 5` exported constant** with a runtime `console.warn`
   if `heroCarouselTools.length` ever exceeds it. The CSS `hcCardCycle`
   keyframes are hand-tuned for exactly this count; the constant + warning
   makes the invariant explicit so a future refactor can't unknowingly
   regress back into the "red/orange overlay bleed-through" bug.

2. **Counter hardening.** Replaced template math
   `0{{ i + 1 }}/0{{ heroCarouselTools.length }}` (which would render as
   `010/010` the moment the count crossed 10 — exactly the class of silent
   regression that produced `01/014` in the original bug report) with a
   precomputed `HeroCarouselCard` view model using `padStart(2, '0')`.
   Separator is now `01 / 05` with explicit spaces + an inline HTML comment
   flagging that the field is a card index, not an `mm:ss` timer.

3. **A11y + ghost-click defense.** `aria-hidden="true"` on the decorative
   rotating carousel (screen readers no longer announce cycling previews)
   and `pointer-events: none` on `.hc-card` so invisible fading slots can't
   catch ghost clicks or tab focus during transitions.
